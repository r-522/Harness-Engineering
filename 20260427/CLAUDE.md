# CLAUDE.md - Claude Code運用規約

このファイルはClaude Codeプロジェクトの動作制約、コーディング規約、セキュリティ要件を定義します。

## 環境設定

### 推奨モデル
```json
{
  "default_model": "claude-opus-4.7",
  "fast_mode_model": "claude-sonnet-4.6",
  "light_tasks_model": "claude-haiku-4-5-20251001",
  "reasoning_model": "claude-opus-4.7-thinking"
}
```

### コンテキスト管理戦略
- **最大コンテキスト使用率**: 60% (フル充填時のパフォーマンス低下を回避)
- **自動サブエージェント分離トリガー**: コンテキスト > 50%
- **キャッシング優先**: システムプロンプト、共通ライブラリ、頻出データ
- **TTL設定**: 明示的に 60分に設定（デフォルト5分を避ける）

## コーディング規約

### 原則
1. **最小限のコード**
   - できるだけシンプルに。複雑さが追加価値を生むまで待つ
   - 削除できるコードは追加より優先
   - 3行重複したら抽象化を検討（それ以前は許容）

2. **エラーハンドリング**
   - 起こり得ないシナリオに防衛的プログラミングをしない
   - システムバウンダリ（入力、外部API）でのみバリデーション
   - 内部コードはフレームワークの保証を信頼

3. **セキュリティ最優先**
   - インジェクション脆弱性（SQL、XSS、コマンド）のチェックは必須
   - 認証・認可フローの検証
   - シークレット・認証情報は `.env`、`.gitignore`対象

4. **非同期・ストリーミング対応**
   - ロング実行タスクは `Bash` tool の `run_in_background` を活用
   - `Monitor` tool でストリーミング監視
   - コンテキスト保全のため適時クリーンアップ

## ツール利用ガイドライン

### 推奨使用順序
```
Read (ファイル読込) 
  → Edit/Write (編集) 
    → Bash (実行) 
      → Agent (複雑なタスク委譲)
```

### ツール別制約

| ツール | 用途 | 制約 | TTL |
|-------|------|------|-----|
| **Read** | ファイル読込 | 最初の1回は必須。その後は編集前に再度読込 | なし |
| **Edit** | 既存ファイル修正 | `old_string` がユニークであること | なし |
| **Write** | 新規ファイル作成 | 既存ファイルは `Edit` を優先 | なし |
| **Bash** | CLI実行 | `cd` は避け絶対パス使用。`cat`, `echo` より `Read`/`Write` | なし |
| **Agent** | タスク委譲 | 複数並列呼び出し時は依存関係確認 | なし |
| **WebSearch** | Web検索 | 認証不要なURL限定。プライベートURLは対象外 | 24h |

## Git運用規約

### ブランチ戦略
- **開発ブランチ**: `claude/*` プリフィックス（自動生成）
- **ベースブランチ**: 明示的に指定されない場合は `main`
- **強制プッシュ禁止**: `--force`, `--force-with-lease` は避ける

### コミットメッセージ形式
```
<type>: <subject>

<body>

<footer>
```

**Type**: `feat`, `fix`, `docs`, `refactor`, `test`, `perf`, `chore`
**Subject**: 命令形、末尾にピリオドなし、50文字以下
**Body**: 何をしたかではなく、**なぜ**を説明
**Footer**: Issue番号、Breaking Changes記載

### コミット前チェック
- [ ] テスト実行確認
- [ ] セキュリティレビュー対象か確認
- [ ] 未コミット機密情報がないか確認（`git status`）
- [ ] `.env`, `credentials.json` が `.gitignore` に登録されているか

## セキュリティチェックリスト

すべてのコード変更前に以下を確認：

### インジェクション脆弱性
- [ ] SQLクエリは必ずパラメータ化クエリ化
- [ ] ユーザー入力をシェルコマンドに直接渡していないか
- [ ] HTMLテンプレートで適切にエスケープされているか

### 認証・認可
- [ ] 認証トークンはセキュアに管理されているか
- [ ] エンドポイントのアクセス制御は実装されているか
- [ ] セッション管理は安全か

### データハンドリング
- [ ] 個人情報（PII）は暗号化されているか
- [ ] ログ出力に機密情報が含まれていないか
- [ ] APIレスポンスは最小限のデータのみ返却か

### 依存関係
- [ ] 既知の脆弱性を持つライブラリが導入されていないか
- [ ] 非公式・未検証ソースのライブラリは避けているか

## スキル定義テンプレート

新規スキル追加時のテンプレート：

```yaml
name: skill-name
trigger: "pattern-to-match"
automation: true
description: "何をするスキルか"
fields:
  - name: input_field
    type: string
    required: true
examples:
  - prompt: "例示的なプロンプト"
    expected_output: "期待される出力"
```

スキルは **週1回以上の頻度で発動するタスク** に限定。一度きりの作業には使用しない。

## パフォーマンス最適化指針

### コンテキスト圧縮
- 古い会話は自動圧縮（200メッセージ以上）
- 不要なツール結果は明示的にクリア
- ファイル読込は必要な行数のみ（`limit`, `offset` 活用）

### 並列処理
- 依存関係なしの複数ツール呼び出し → 同時実行
- 依存関係ありの場合 → 逐次実行
- 時間のかかるタスク → `run_in_background` で非ブロッキング化

### キャッシング戦略
```
システムプロンプト (大規模, 頻出) → 最優先キャッシュ
共通ライブラリ定義 → 次優先
ユーザー入力 → キャッシュ対象外（変動高）
```

## トラブルシューティング

### コンテキスト満杯時
1. `Bash` で古いログ削除
2. `Agent` にタスク委譲（新規コンテキスト取得）
3. ツール結果の明示的クリア

### ツール実行失敗時
1. 原因を診断（ネットワーク, パーミッション, 入力値）
2. 根本原因を修正（回避策でなく）
3. 必要に応じて再試行（指数バックオフ）
