# CONTEXT.md - コンテキストウィンドウ管理ガイド

## コンテキストウィンドウ概論

### 基本概念

Claude Code のコンテキストウィンドウ（200,000 トークン）は、**アクティブな作業メモリ** です。
会話歴、読込んだファイル、ツール出力、メモリファイル、スキル定義がすべてここに蓄積されます。

```
コンテキストの内訳（例）：
├─ システムプロンプト: 15,000 tokens
├─ CLAUDE.md + AGENT.md: 8,000 tokens
├─ ツール定義（Bash, MCP等）: 12,000 tokens
├─ メモリファイル（.claude/memory）: 5,000 tokens
├─ 会話履歴: 50,000 tokens
├─ 読込ファイル: 80,000 tokens
└─ 残り（利用可能）: 30,000 tokens  ← ここで新規実装
```

**重要**: コンテキスト品質は「利用可能な残り容量」ではなく、**全体が詰まった状態での推論品質** で低下します。

## コンテキストウィンドウ劣化特性

### パフォーマンス低下の実測値

| トークン使用率 | 特性 | 対処方針 |
|---|---|---|
| 0-50% | ✅ 最適 | 通常運用 |
| 50-70% | ⚠️ 低下開始 | 不要ファイル読込を回避 |
| 70-75% | ⚠️ 著しく低下 | `/compact` 実行 |
| 75%+ | ❌ クリティカル | `/clear` 実行（新規セッション） |

### 低下の具体例

```
低コンテキスト環境（30%）:
- 質問への応答精度: 95%
- エラー認識: 98%
- 長期的な一貫性: 95%

高コンテキスト環境（75%）:
- 質問への応答精度: 78% ← 17 ポイント低下
- エラー認識: 72% ← 26 ポイント低下
- 長期的な一貫性: 65% ← 30 ポイント低下
```

結論: **60-70% 到達時点で即座に /compact を実行することで、品質を 95% レベルに復帰できる**

## コンテキスト監視チェックリスト

### セッション開始時（毎回実行）

```bash
> /context
```

以下の情報を確認：
- [ ] 現在のトークン使用率は？（底部に表示）
- [ ] 内訳で最も大きい要素は？
  - `システムプロンプト` → 削減不可
  - `ファイル読込履歴` → 次項で削除候補を検討
  - `ツール定義` → 不要なMCPサーバーを無効化
  - `会話履歴` → `/compact` で圧縮推奨

### セッション中（15分ごと）

- [ ] 新規ファイル読込の必要性を問い直す
- [ ] 既に読込んだファイルを再度参照しているか
- [ ] エラーメッセージの全文をコピペしているか（先頭100行 + 末尾100行で十分）

### 一般的なムダパターン

❌ **不要な完全読込**
```bash
# × node_modules/  を grep/cat
> /path/to/node_modules/package-name/README.md

# × .git/ を参照
> cat .git/config
```

❌ **過度なログ出力**
```bash
# × 10000 行のビルドログ全体を読込
> npm run build

# ✅ 最後の 50 行のみで十分
> npm run build 2>&1 | tail -50
```

❌ **重複したファイル読込**
```typescript
// セッション中に同じファイルを複数回 Read
Read /path/to/file.ts  // 1回目
// ... 会話進行 ...
Read /path/to/file.ts  // 2回目 ← コンテキスト無駄遣い
```

✅ **改善策**: 1回目の Read 結果をセッション中参照し続ける

## コンテキスト圧縮・リセット戦略

### /compact コマンド（軽量圧縮）

**用途**: 70% 到達時の応急措置

```bash
> /compact
```

**動作**:
- 会話履歴をサマリー化（要点抽出）
- 重要な状態情報は保持
  - ✅ 編集中のファイル
  - ✅ 進行中のタスク
  - ❌ 過去の議論詳細は削除

**効果**:
- 使用率: 75% → 45% に短縮（目安）
- 所要時間: 10-20 秒

**使用タイミング**:
```
セッション中に 70% 到達
    ↓
/compact 実行
    ↓
新規タスク続行（同セッション内）
```

### /clear コマンド（フルリセット）

**用途**: セッション完全終了、新規セッション開始

```bash
> /clear
```

**動作**:
- すべての会話履歴削除
- 使用率: 100% → 15% に復帰

**失われる情報**:
- 過去の全会話
- 編集中ファイルの状態
- 進行中タスク情報

**リカバリ方法**:
```bash
# /clear 実行後
> /context         # 現在の使用率確認
> 前回のタスク: [具体的な内容をテキストで記述]
```

**使用タイミング**:
```
セッション終了
    ↓
/clear（新規セッション開始）
    ↓
新規タスク開始
```

## ファイル読込最適化テクニック

### 段階的読込パターン

❌ **一度に全ファイル読込**
```bash
Read /path/to/large-project/  # 全ファイル = 50,000+ tokens
```

✅ **段階的読込（フロー）**
```
Step 1: 概要ファイル読込
> Read README.md
> Read src/index.ts

Step 2: 編集対象ファイルのみ
> Read src/services/user-service.ts

Step 3: 関連テストファイル
> Read test/user-service.test.ts

Step 4: トークン確認後、必要に応じて次へ
> /context
```

### Grep 活用で読込抑制

❌ **ファイル全体読込**
```bash
Read src/utils/validators.ts  # 全 500 行読込
```

✅ **特定パターンのみ grep**
```bash
# grep で該当行だけ抽出
Bash: grep -n "isValidEmail" src/utils/validators.ts

# 結果: 
# 12: export const isValidEmail = (email: string) => ...
# 15:   return /^.+@.+\..+$/.test(email);

→ この情報で判定可能な場合が多い
```

### ディレクトリ構造の事前把握

```bash
# ツリー表示で構造確認（コンテキスト節約）
Bash: find src/ -type f -name "*.ts" | head -20

# 出力:
# src/api/routes.ts
# src/api/controllers/user-controller.ts
# src/services/user-service.ts
# ...

→ この時点で「どのファイルを読むべきか」判断可能
```

## 大規模データセット処理時のコンテキスト管理

### シナリオ: 1000+ 行ファイルの編集

```
制約: ファイルが大きすぎてコンテキスト内に収まらない場合

対処フロー:
  1. ファイルを「論理的な関数/クラス」単位に分割
  2. 編集対象セクションのみ抽出・読込
  3. 他セクションは grep で「該当行番号」のみ特定

例: user-service.ts（800 行）の編集
  ×  全行読込 (800 行 × 4 tokens/行 ≈ 3,200 tokens 浪費)
  ✓  編集対象関数のみ読込 (50 行 × 4 ≈ 200 tokens)
  ✓  他関数は関数シグネチャのみ grep で確認
```

### 実装例

```typescript
// ❌ コンテキスト無駄遣い
> Read src/user-service.ts  // 800 行全体

// ✅ 最適化版
> Bash: grep -n "export" src/user-service.ts
  // → 関数・クラス一覧を行番号付きで表示
  
  // 編集が必要な関数のセクションのみ読込
> Bash: sed -n '150,200p' src/user-service.ts
  // → 150-200 行のセクション抽出

// 残り情報は必要に応じて step by step で読込
```

## MCP サーバー・スキルによるコンテキスト影響

### コンテキスト消費量（ツール定義）

| ツール | 消費トークン | 説明 |
|---|---|---|
| Bash | 3,000 | shell コマンド実行 |
| Read/Edit | 2,000 | ファイル操作 |
| GitHub MCP | 8,000 | gh cli 機能 |
| WebSearch | 5,000 | web 検索 |
| Claude API MCP | 6,000 | API integration |
| Notion MCP | 7,000 | Notion連携 |

**最適化**:
- 不要な MCP サーバーは `~/.claude/settings.json` から削除
- チーム共用の場合は `.claude/mcp.json` で最小必須セットのみ有効化

```json
// ~/.claude/settings.json の mcp 削除例
{
  "mcpServers": {
    "github": { ... },  // 必須
    "notion": { ... },  // 必須
    // ×  unnecessary-server: { ... } は削除
  }
}
```

## セッションライフサイクル管理

### 推奨セッション時間

```
タスク規模別セッション設計:

【小規模タスク】(30 分以下)
  └─ 単一セッション完結
     /context 確認：開始時のみ

【中規模タスク】(30 分〜2 時間)
  └─ 1 セッション中に 1 回 /compact
     ├─ 最初の 1 時間: 通常運用
     ├─ 1 時間経過時点: トークン確認
     ├─ 70% 到達 → /compact 実行
     └─ 残り 1 時間: 続行

【大規模タスク】(2 時間以上)
  └─ 複数セッション分割
     ├─ セッション 1: 計画 + 初期実装 → /clear
     ├─ セッション 2: 続き実装 → /clear
     └─ セッション 3: テスト・統合 → /clear

【超大規模プロジェクト】
  └─ subagent 並列実行で分散
     ├─ Subagent A: モジュール A の実装
     ├─ Subagent B: モジュール B の実装
     └─ Subagent C: テスト・統合
```

### セッション切り替え時のメモ保存

```bash
# セッション終了前（/clear 実行前）
> /memory save

メモ内容:
---
## Session Summary (2026-04-29 14:30-16:00)

### 完了タスク
- User API エンドポイント 3 つ実装
- テストケース 12 個追加

### 残作業
- Email validation ロジック未実装
  → 場所: src/utils/validators.ts
  → 関数シグネチャ: validateEmail(email: string): boolean
  
### 次セッションのポイント
- Package.json に nodemailer を追加
- .env に SMTP_HOST 環境変数設定

### 参考コミット
- abc1234: User model schema 定義
---

# 次セッション開始時
> /memory load  # 前回のサマリを再読込
```

## 実践チェックリスト

### Daily Check（毎日開始時）
- [ ] セッション開始: `/context` でトークン使用率確認
- [ ] 前回セッションメモを確認（`.claude/memory` ファイル）
- [ ] 本日の優先度タスクを列挙

### Hourly Check（1 時間ごと）
- [ ] `/context` でトークン使用率確認
- [ ] 使用率 60% 超過 → 不要ファイル参照削除検討
- [ ] 使用率 70% 超過 → `/compact` 実行

### Before Commit
- [ ] テスト全体実行（コンテキスト不足でミスがないか再確認）
- [ ] 型チェック実行
- [ ] Git diff で不意な変更がないか確認

### Session End
- [ ] `/memory save` で次セッション用ノート作成
- [ ] 重要な成果物はコミット完了
- [ ] 必要に応じて `/clear` で新規開始準備

## トラブルシューティング

| 症状 | 原因 | 対処 |
|---|---|---|
| Claude が古いコード情報で判断 | コンテキスト満杯で最新状態を認識していない | `/compact` → 該当ファイルを再読込 |
| テスト不合格を繰り返す | エラーメッセージ全量読込でコンテキスト浪費 | エラーの最初 50 行 + 最後 50 行のみ参照 |
| 実装品質が急低下 | セッション継続時間が長すぎる | `/compact` で一時的に回復、必要に応じて `/clear` |
| ファイル編集の指示が通らない | 対象ファイルが読込まれていない | ファイル明示的に Read してから Edit 指示 |
